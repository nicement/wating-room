# PRD: 대기열 시스템 (Queueing Solution)

## 1. 개요 (Overview)
본 문서는 트래픽 폭주 시 백엔드 시스템을 보호하고 사용자에게는 순차적인 접속을 보장하는 **'대기열 시스템'**에 대한 제품 요구사항 정의서(PRD)입니다. 사용자가 서비스에 진입할 때 허용된 접속 가능 인원을 초과한 상태라면 대기열로 안내되며, 지속적인 상태 갱신(Polling)을 통해 자신의 순서를 확인합니다. 순서가 되면 자동으로 목적지 시스템으로 접속할 수 있게 지원합니다.

## 2. 목적 (Objectives)
- **시스템 보호**: 갑작스러운 트래픽 급증(Spike)으로부터 백엔드(목적지) 시스템의 서버 과부하 및 다운타임 방지
- **사용자 경험(UX) 개선**: 무응답 화면이나 에러 메세지 대신, 명확한 대기 순번 및 진행 상황을 안내하여 사용자의 이탈률 최소화
- **공정성 보장**: 먼저 진입한 사용자가 먼저 서비스를 이용할 수 있도록 선착순(FIFO) 기반의 공정한 접속 순서 보장

## 3. 핵심 기능 요구사항 (Functional Requirements)

### 3.1. 진입 및 동시 접속량 제어 (Entry & Capacity Control)
- **동시 접속자 수 관리**: 목적지 시스템의 최대 허용 동시 작업 수(Capacity)를 설정할 수 있어야 합니다.
- **진입 제어 로직**: 
  - 현재 작업 중인 활성 사용자 수가 Capacity 미만인 경우: 사용자는 대기 없이 즉시 목적지 시스템으로 진입(Bypass)합니다.
  - 현재 작업 중인 활성 사용자 수가 Capacity 이상인 경우: 사용자는 대기열 큐(Queue)로 편입됩니다.

### 3.2. 대기열 상태 안내 (Queue Status & UI)
- **대기 번호(티켓) 부여**: 대기열 진입 시 고유한 세션(또는 티켓)이 사용자 브라우저에 배정됩니다.
- **실시간 대기 순번 화면 제공**: 
  - 현재 자신이 대기열 중 몇 번째인지 명확하게 보여주는 UI를 제공해야 합니다.
  - (선택 사항) 현재 처리 속도를 바탕으로 한 예상 대기 시간을 함께 표시합니다.

### 3.3. 순서 갱신 (Polling)
- **주기적 상태 확인**: 클라이언트(브라우저)는 주기적(예: 3~5초 간격)으로 대기열 서버에 API 요청(Polling)을 보내어 자신의 최신 대기 순번을 받아옵니다.
- **비정상 연결 관리**: 일정 시간 이상 Polling 요청이 없는 클라이언트는 이탈로 간주하여 대기열에서 자동으로 제거해야 합니다.

### 3.4. 연결된 시스템으로의 접속 (Redirection)
- **순번 도달 감지**: 사용자의 차례가 오면, Polling API 응답으로 사용자에게 진입이 가능함을 알리고 일회성 접속 토큰(또는 세션 키)을 반환합니다.
- **자동 접속 처리**: 클라이언트는 진입 가능 응답을 받는 즉시, 발급받은 토큰과 함께 목적지 시스템 URL로 자동 리다이렉트(Redirect) 합니다.
- **안전한 진입 통제**: 목적지 시스템은 대기열을 거치치 않은 비정상적인 우회 접속을 방지하기 위해 토큰의 유효성을 검증해야 합니다.

## 4. 비기능 요구사항 (Non-Functional Requirements)
- **성능 (Performance)**: 수많은 사용자가 동시에 Polling 요청을 보내므로, 서버는 매우 가볍고 빠르게(응답시간 최소화) 상태를 반환해야 합니다. (Redis 등 인메모리 데이터 스토어 활용 권장)
- **확장성 (Scalability)**: 일시적인 대규모 트래픽을 무리 없이 수용할 수 있도록 대기열 시스템 자체가 쉽게 스케일 아웃(Scale-out) 가능한 구조여야 합니다.
- **신뢰성 (Reliability)**: 큐 시스템의 일부 모듈이 재시작되더라도 이미 확보된 사용자의 대기 순서가 뒤섞이거나 유실되어서는 안 됩니다.

## 5. 시스템 흐름도 (System Flow)
1. 사용자가 서비스 도메인으로 접속 요청
2. **[대기열 시스템]** 유효한 세션 확인 및 활성 사용자 수 확인
3. **[분기]**
   - **(3-A) 허용 인원 이내**: 진입 토큰 발급 및 목적지 도메인으로 Redirect ➡️ **(프로세스 종료)**
   - **(3-B) 허용 인원 초과**: 대기열 Queue에 등록하고 대기 번호 부여
4. **[클라이언트]** 대기 번호를 기반으로 대기 안내 화면을 로드
5. **[클라이언트 🔄 대기열 시스템]** 주기적으로 Polling API를 호출하여 현재 대기 순서를 요청
6. **[대기열 시스템]** 목적지 시스템의 활성 사용자 수가 감소(세션 만료 등)하여 자리가 나면 대기열 맨 앞 사용자의 상태를 '진입 가능'으로 변경
7. **[클라이언트]** 다음 Polling 시 '진입 가능' 상태와 '접속 토큰'을 응답받음
8. **[클라이언트]** 접속 토큰을 가지고 **[목적지 시스템]**으로 리다이렉트 (연결)
9. **[목적지 시스템]** 토큰 유효성 검증 후 서비스 제공

## 6. 향후 고려사항 (Out of Scope / Future Work)
- WebSocket 또는 SSE(Server-Sent Events)를 활용한 양방향 통신 (현재는 단순 구현을 위해 Polling 선택)
- 특정 사용자 군(VIP 등)을 위한 패스트트랙(Fast-track) 대기열 분리 기능
